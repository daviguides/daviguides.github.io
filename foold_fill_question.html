ASCII Paint Bucket¶

Your friend is developing a small image processing program and has asked for your help in implementing MS-Paint's paint bucket-like functionality.

Their program represents images using arrays of characters, with each array value representing a pixel and letters and symbols representing different colors.

For example, the following 4x6 matrix represents the letter P in color "#", with background color "." (dot)

.###..
.#..#.
.###..
.#....

Your subroutine should:

  1. take a pixel and a new color
  2. and paint the region of that pixel with the new color,

like MS-Paint's paint bucket tool.

First Solution¶

We can using DFS with recursion as below:

In [ ]:
def flood_fill_dfs(image_grid, x, y, new_color):
    rows, cols = len(image_grid), len(image_grid[0])
    old_color = image_grid[x][y]

    def dfs(x, y):
        if x < 0 or x >= rows or y < 0 or y >= cols:
            return
        if image_grid[x][y] != old_color:
            return
        image_grid[x][y] = new_color

        dfs(x + 1, y)
        dfs(x - 1, y)
        dfs(x, y + 1)
        dfs(x, y - 1)

    dfs(x, y)

    return image_grid

Other solutions using: Optimized Depth-First Search and Breadth-First Search¶

Taking a step back, the first thing we have to deal with is the best-case scenario O(1), if the pixel's old_color is equal new_color does nothing and the function returns.

Of course this solves only the happiest case, and to solve the other cases we will use two options of graph traversal algorithms:

  • Depth-First Search (DFS)
  • Breadth-First Search (BFS)

So for simplicity our first function will receive an additional argument called gt_algorithm which is a function that will be called internally:

In [ ]:
def flood_fill(grid, x, y, new_color, gt_algorithm):
    old_color = grid[x][y]
    if new_color == old_color:
        return
    gt_algorithm(grid, x, y, old_color, new_color)

Using Depth-First Search option with stack to avoid recursion and stack overflow errors¶

DFS with stack¶

To avoid the recursion error we can implement dfs using a list as a stack (LIFO)

In [ ]:
# Function using depth-first search (DFS) using stack
def dfs_with_stack(grid, x, y, old_color, new_color):
    rows, cols = len(grid), len(grid[0])
    stack = [(x, y)]
    while stack:
        # LIFO
        x, y = stack.pop()
        if 0 <= x < rows and 0 <= y < cols\
                and grid[x][y] == old_color:
            grid[x][y] = new_color
            stack.extend([
                (x + 1, y), 
                (x - 1, y), 
                (x, y + 1), 
                (x, y - 1)])

    return grid

# Function using breadth-first search (BFS) using list queue
def bfs_with_list_as_queue(grid, x, y, old_color, new_color):
    rows, cols = len(grid), len(grid[0])

    queue = [(x, y)]
    while queue:
        # FIFO
        x, y = queue.pop(0)
        if 0 <= x < rows and 0 <= y < cols \
                and grid[x][y] == old_color:
            grid[x][y] = new_color
            queue.extend([
                (x + 1, y), 
                (x - 1, y), 
                (x, y + 1), 
                (x, y - 1)])
    return grid

Using deque no instead of list¶

But for performance terms it is better to use what as queue (FIFO)

In [ ]:
from collections import deque

# Function using breadth-first search (BFS) using deque as queue
def bfs_with_deque_as_queue(grid, x, y, old_color, new_color):
    rows, cols = len(grid), len(grid[0])

    queue = deque([(x, y)])
    while queue:
        x, y = queue.popleft()
        if 0 <= x < rows and 0 <= y < cols \
                and grid[x][y] == old_color:
            grid[x][y] = new_color
            queue.extend([
                (x + 1, y), 
                (x - 1, y), 
                (x, y + 1), 
                (x, y - 1)])

    return grid

Using set for labeling¶

# Function using depth-first search (DFS) using stack
def dfs_with_stack_label(grid, x, y, old_color, new_color):
    rows, cols = len(grid), len(grid[0])
    stack = [(x, y)]
    visited = set()

    while stack:
        # LIFO
        x, y = stack.pop()
        if 0 <= x < rows and 0 <= y < cols\
                and grid[x][y] == old_color:
            visited.add((x, y))
            grid[x][y] = new_color
            stack.extend([
                (x + 1, y), 
                (x - 1, y), 
                (x, y + 1), 
                (x, y - 1)])

    return grid

Using a matrix for labeling¶
In [ ]:
# Function using depth-first search (DFS) using stack
def dfs_with_stack_with_matrix_selected(grid, x, y, old_color, new_color):
    rows, cols = len(grid), len(grid[0])
    stack = [(x, y)]
    visited = [[False]*cols for _ in range(rows)]

    while stack:
        # LIFO
        x, y = stack.pop()
        if 0 <= x < rows and 0 <= y < cols\
                and grid[x][y] == old_color:
            visited[x][y] = True
            grid[x][y] = new_color
            stack.extend([
                (x + 1, y), 
                (x - 1, y), 
                (x, y + 1), 
                (x, y - 1)])

    return grid

Conclusion¶

As observed in the graph, the most performant functions were:

  • bfs_with_deque_as_queue
  • dfs_with_stack

Very close to these we have:

  • dfs_with_stack_and_matrix

And the least performant of these options were:

  • dfs_with_stack_label
  • bfs_with_list_as_queue

Among the given functions, the one with the best performance in terms of time complexity would be bfs_with_deque_as_queue using a deque as a queue. This is because a deque provides efficient operations for adding elements to the end (queue.extend) and removing elements from the front (queue.popleft). These operations have a time complexity of O(1) on average.

On the other hand, dfs_with_stack and bfs_with_list_as_queue that use a list as a queue, which requires shifting all elements after a drop operation, resulting in a time complexity of O(n) for each drop. Although using list to queue bfs_with_list_as_queue ends up making it the slowest of all.

Already in the functions dfs_with_stack_label and dfs_with_stack_with_matrix_selected which should be improved versions of dfs_with_stack. We can see that our labeling system that we intended to use, ended up bringing more cost than performance increase, as they only add additional operations to track visited cells.

So overall, using a deque as a queue in bfs_with_deque_as_queue provides the best performance among the given functions due to efficient removal of elements from the front, resulting in the best optimal timing complexity.
